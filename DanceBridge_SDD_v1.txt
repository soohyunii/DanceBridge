DANCEBRIDGE SOFTWARE DESIGN DOCUMENT (SDD)
Version: 1.0
Target: Web + iOS + Android
Document Type: MVP Implementation Guide


0. DOCUMENT PURPOSE

이 문서는 DanceBridge MVP를 구현할 개발자가 전체 시스템 구조, 핵심 도메인, 데이터 모델, API, 화면 흐름, 권한, 검색 및 배포 방식을 빠르게 이해하고 개발을 시작할 수 있도록 작성한 Software Design Document이다.

DanceBridge는 수강생과 댄서를 연결하는 양면 마켓플레이스다.

핵심 가치:
- 수강생: 클래스의 분위기와 심리적 진입 장벽을 확인하고 자신에게 맞는 수업을 검색·신청한다.
- 댄서: 자신의 장르, 스타일, 무드, 영상, 수업 방식을 구조화하여 등록하고 더 정확한 수강생에게 발견된다.

MVP에서는 AI 자동 분석보다 명시적 태그와 구조화된 데이터를 우선한다.

3. USER AND AUTHORIZATION MODEL

3.1 역할

USER_ROLE
- STUDENT
- DANCER
- ADMIN

하나의 User가 MVP에서는 하나의 기본 역할을 가진다.
향후 동일 계정에서 STUDENT와 DANCER 역할을 동시에 사용할 수 있도록 UserRole 조인 테이블로 확장 가능하게 설계하는 것이 좋다.

3.2 인증 방식

지원 로그인
- Email + Password
- Google OAuth
- Apple Sign In
- Kakao OAuth는 한국 시장 검증 후 추가 가능

토큰
- Access Token: 15분
- Refresh Token: 14~30일
- Refresh Token Rotation 적용
- 모바일은 Secure Storage, 웹은 HttpOnly Secure Cookie 권장

3.3 권한 규칙

STUDENT
- 공개 댄서 및 클래스 조회
- 찜, 신청, 취소, 리뷰 작성
- 자신의 신청 내역 조회

DANCER
- 자신의 댄서 프로필 수정
- 자신의 영상과 클래스 CRUD
- 자신의 클래스 신청자 조회
- 신청 상태 변경

ADMIN
- 모든 리소스 조회
- 검수, 숨김, 정지, 삭제
- 메타데이터 관리

모든 수정 API는 리소스 소유권 검증이 필요하다.

7. SEARCH DESIGN

7.1 MVP 검색

PostgreSQL 기반
- pg_trgm
- Full Text Search
- ILIKE는 소규모 데이터에서만 사용
- normalizedTitle, stageName, class title에 검색용 인덱스 적용

검색 우선순위 예시
1. 정확한 노래 제목 일치
2. 댄서명 정확 일치
3. 클래스 제목 일치
4. 장르 및 무드 태그
5. 설명문 검색

7.2 Song Search Flow

사용자가 “열이올라요” 검색
1. Song.normalizedTitle 검색
2. 매칭 Song ID 조회
3. 해당 Song과 연결된 DanceVideo 조회
4. DanceVideo의 DancerProfile 조회
5. 해당 댄서가 운영하는 공개 Class 조회
6. 장르 및 무드 필터 적용
7. 영상 카드 중심으로 결과 노출

검색 결과 구조
Song
  -> Videos
      -> Dancer
          -> Available Classes

7.3 Mood Search Flow

1. Mood 태그 검색
2. DanceVideoMood 조회
3. 관련 영상과 댄서 계산
4. 해당 Mood를 가진 클래스 조회
5. 사용자 경험 수준과 지역에 따라 정렬

7.4 추천 점수 MVP

규칙 기반 점수 사용

score =
  genreMatch * 30
+ moodMatch * 20
+ experienceMatch * 20
+ regionMatch * 10
+ scheduleMatch * 10
+ beginnerFriendlyMatch * 5
+ languageMatch * 5

정확한 가중치는 운영 데이터로 조정한다.
개인화 모델은 충분한 클릭·찜·예약 데이터가 쌓인 후 도입한다.


8. REVIEW AGGREGATION

리뷰 데이터는 Class와 Dancer 프로필에 집계하여 제공한다.

예시 집계 필드
- averageOverallRating
- averageActualDifficulty
- averagePsychologicalEntryDifficulty
- averageBeginnerFriendly
- averageSoloFriendly
- averageExplanationClarity
- averageRepetitionLevel
- averageEnergyLevel
- touchPolicyConsistency

집계 방식
- 리뷰 생성 또는 수정 시 Queue에 Aggregate Job 등록
- Worker가 집계 테이블 업데이트
- 조회 시 매번 AVG 연산하지 않도록 별도 Summary Table 사용 가능

ReviewSummary
- targetType: CLASS | DANCER
- targetId
- reviewCount
- aggregated JSON
- updatedAt

리뷰 수가 적을 때 과도한 확신을 주지 않도록 sampleCount를 화면에 함께 표시한다.

11. MEDIA PROCESSING

11.1 업로드 과정

1. Client가 POST /media/upload-url 호출
2. Backend가 Pre-signed Upload URL 반환
3. Client가 Storage에 직접 업로드
4. Client가 업로드 완료 API 호출
5. Backend가 MediaAsset 상태를 PROCESSING으로 설정
6. Queue Worker가 FFmpeg 처리
7. 썸네일 생성 및 메타데이터 추출
8. READY 상태로 변경
9. DanceVideo 생성 가능

11.2 권장 제한

- 영상 길이: 최대 90초
- 파일 크기: 최대 300MB
- 허용 포맷: MP4, MOV
- 출력 포맷: H.264 MP4
- 썸네일: WebP
- 이미지: JPEG, PNG, WebP

11.3 주의사항

외부 Instagram Reel URL만 저장하는 방식은 삭제, 권한, 임베드 정책 변경 위험이 있다.
MVP에서는 사용자가 직접 업로드하거나 자신이 소유한 콘텐츠 링크를 등록하도록 한다.

13. PAYMENT DESIGN

MVP 선택지 A
- 플랫폼 외부 예약 링크
- 결제 구현 없음
- 가장 빠른 검증 가능

MVP 선택지 B
- 플랫폼 내부 신청만 관리
- 현장 결제
- paymentStatus = NOT_REQUIRED

MVP 선택지 C
- 국내 결제 연동
- Toss Payments 또는 PortOne
- 결제 Webhook 필요
- 멱등성 키 적용
- Webhook Signature 검증

초기에는 A 또는 B로 시작하고, 예약 전환율이 검증된 후 C를 구현하는 것을 권장한다.

21. MVP ACCEPTANCE CRITERIA

다음 조건을 만족하면 MVP 출시 가능 상태로 판단한다.

- 학생과 댄서가 서로 다른 온보딩을 완료할 수 있다.
- 승인된 댄서만 공개 프로필과 클래스를 게시할 수 있다.
- 댄서가 영상, 노래, 장르, 무드 정보를 등록할 수 있다.
- 학생이 노래, 장르, 무드, 지역, 수준으로 검색할 수 있다.
- 학생이 클래스 세션에 신청할 수 있다.
- 정원을 초과한 예약이 생성되지 않는다.
- 댄서가 신청을 승인 또는 거절할 수 있다.
- 완료된 예약에 대해서만 리뷰를 작성할 수 있다.
- 클래스 상세에서 심리적 진입 관련 리뷰 집계가 표시된다.
- 웹과 모바일이 동일 API와 동일 계정 데이터를 사용한다.
- 관리자에게 댄서 검수와 신고 처리 기능이 제공된다.

23. FINAL IMPLEMENTATION SUMMARY

개발 관점에서 DanceBridge의 핵심은 다음 네 가지 관계를 안정적으로 구현하는 것이다.

1. Dancer -> DanceVideo -> Song / Genre / Mood
2. Dancer -> Class -> ClassSession
3. Student -> Booking -> Review
4. Search Query -> Video / Dancer / Class Result

MVP는 AI 플랫폼이 아니라 구조화된 마켓플레이스다.
처음부터 자동 영상 분석이나 복잡한 추천 모델을 구현하지 않는다.

우선 사용자가 직접 입력한 태그와 구조화된 리뷰를 사용해
- 수강생이 자신에게 맞는 클래스를 찾을 수 있는지
- 댄서가 새로운 수강생에게 발견되는지
- 노래 검색이 실제 예약으로 연결되는지

세 가지를 검증한다.

검증 후 AI, 결제, 개인화 추천, 공연 및 프로젝트 매칭으로 확장한다.
